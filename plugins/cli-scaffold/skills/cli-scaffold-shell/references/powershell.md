# PowerShell CLI generation reference

Pin every PowerShell CLI generated by this skill to the idioms below. Do not
invent alternative shapes — regenerate the same structure every run.

## 1. Framework

Default to native `[CmdletBinding()]` + `param()` blocks. Never add a
third-party argument-parsing library — PowerShell's own parameter binder
already is the declarative parser: it derives type coercion, validation,
mandatory-ness, and `--help`/`Get-Help` surfacing from the attributes below
with zero extra dependencies.

Minimal invocation shape:

```powershell
function Get-Greeting {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory = $true, Position = 0)]
        [string]$Name,

        [Parameter()]
        [switch]$Shout
    )
    ...
}
```

Always attach `[CmdletBinding()]` — it enables common parameters
(`-Verbose`, `-ErrorAction`, `-WhatIf`) and `Get-Help` integration even for
a function with zero declared parameters. Always type every parameter
(`[string]`, `[int]`, `[switch]`, `[string[]]`) — untyped parameters lose
automatic validation and produce weaker `Get-Help` output. Mark a
parameter `[Parameter(Mandatory = $true)]` rather than hand-writing a null
check; the parameter binder prompts interactively for it if omitted (guard
against that in non-interactive mode — see Section 6).

Name every generated function with the PowerShell approved verb-noun
convention: `<ApprovedVerb>-<Noun>`, singular noun, PascalCase both sides
(`Get-Status`, `New-Project`, `Set-Config`, `Invoke-Build`). Run
`Get-Verb` to pick the verb — it is the canonical list of approved verbs
grouped by action (Common, Data, Lifecycle, Diagnostic, ...); never invent
a verb outside it (no `Fetch-`, `Create-`, `Build-` — use `Get-`, `New-`,
`Invoke-` respectively) since an unapproved verb triggers a module-import
warning and breaks discoverability via `Get-Command -Verb`.

## 2. Project layout

Generate this exact tree:

```
MyTool/
├── MyTool.psd1            # module manifest (New-ModuleManifest)
├── MyTool.psm1             # root module: dot-sources Public/ and Private/, exports
├── Public/
│   └── Get-Greeting.ps1    # one exported function per file, same name as file
├── Private/
│   └── Format-Greeting.ps1 # helper functions, never exported
├── Tests/
│   ├── MyTool.Tests.ps1    # Pester specs
│   └── testdata/
│       └── help.golden.txt # stored Get-Help snapshot, see Section 9
└── README.md
```

A generated CLI IS the module — there is no separate thin wrapper script.
Every function in `Public/` is simultaneously the importable API and the
invokable command; state this explicitly in generated comments as
PowerShell's structural difference from every other language this skill
targets (Python, Rust, Go, etc. split a `core` library from a thin `cli`
entry point because their functions aren't directly invocable from a
shell — a PowerShell cmdlet function already is). Do not force an
artificial `core`/`cli` split onto a module; put business logic straight
into the `Public/`/`Private/` functions.

`MyTool.psm1` dot-sources every script under `Public/` and `Private/`,
then exports only the public ones:

```powershell
Get-ChildItem -Path "$PSScriptRoot\Private\*.ps1", "$PSScriptRoot\Public\*.ps1" |
    ForEach-Object { . $_.FullName }

Export-ModuleMember -Function (Get-ChildItem -Path "$PSScriptRoot\Public\*.ps1").BaseName
```

## 3. Help text and completions

Precede every exported function with a comment-based help block —
`Get-Help` parses it automatically with zero extra registration:

```powershell
<#
.SYNOPSIS
    Builds a greeting for the given name.
.PARAMETER Name
    The name to greet.
.PARAMETER Shout
    Emit the greeting in uppercase.
.EXAMPLE
    Get-Greeting -Name "World"
#>
function Get-Greeting {
    [CmdletBinding()]
    param(...)
}
```

Keep `.SYNOPSIS`, one `.PARAMETER` block per declared parameter (name must
match exactly), and at least one `.EXAMPLE`. This block is what `Get-Help
MyTool\Get-Greeting -Full` renders — never hand-write a separate
`--help`/usage string.

For argument completions, use a static `[ValidateSet(...)]` when the
option list is fixed:

```powershell
[Parameter()]
[ValidateSet("json", "text", "table")]
[string]$Format = "text"
```

For a dynamic list (computed at completion time — e.g. names read from a
config file), attach `[ArgumentCompleter({...})]` to the parameter:

```powershell
[Parameter()]
[ArgumentCompleter({
    param($commandName, $parameterName, $wordToComplete, $commandAst, $fakeBoundParameters)
    Get-ChildItem "$HOME\.mytool\profiles" -Name |
        Where-Object { $_ -like "$wordToComplete*" }
})]
[string]$Profile
```

For completions on a parameter you cannot edit directly (a third-party
cmdlet's parameter, or to keep completion logic out of `Public/`),
register the same scriptblock separately with `Register-ArgumentCompleter
-CommandName Get-Greeting -ParameterName Profile -ScriptBlock {...}` from
`MyTool.psm1`'s load path instead of the inline attribute — both idioms
are equivalent at runtime, so use the inline attribute by default and fall
back to `Register-ArgumentCompleter` only when the attribute can't be
placed on the declaration itself.

## 4. NO_COLOR-aware output

Check `$env:NO_COLOR` explicitly before any manual `Write-Host
-ForegroundColor` call — `Write-Host` has no built-in `NO_COLOR` awareness
on its own:

```powershell
if (-not $env:NO_COLOR) {
    Write-Host "Success" -ForegroundColor Green
} else {
    Write-Host "Success"
}
```

On PowerShell 7.2+, prefer gating through `$PSStyle.OutputRendering`
instead of branching every call site — set it once near the entry point:

```powershell
if ($env:NO_COLOR -and $PSStyle) {
    $PSStyle.OutputRendering = [System.Management.Automation.OutputRendering]::PlainText
}
```

`$PSStyle` does not exist on Windows PowerShell 5.1, so always guard its
use behind `if ($PSStyle)` (or `$PSVersionTable.PSVersion.Major -ge 7`)
and keep the manual `$env:NO_COLOR` check on individual `-ForegroundColor`
calls as the compatibility fallback for 5.1 hosts.

## 5. Exit codes

Frozen cross-paradigm contract — every generated CLI, regardless of
language, must honor this:

- `0` — success
- `1` — general/runtime error (caught error, failed operation)
- `2` — usage/argument error (bad flags, missing required argument)

Use `exit(N)` in a normal console host. Inside hosted contexts where
`exit` doesn't terminate the process as expected (ISE, some remoting
hosts), use `$host.SetShouldExit(N)` instead — detect via
`$host.Name -eq "ConsoleHost"` and prefer `exit` there, falling back to
`$host.SetShouldExit(N)` otherwise:

```powershell
function Exit-WithCode([int]$Code) {
    if ($host.Name -eq "ConsoleHost") { exit $Code }
    else { $host.SetShouldExit($Code) }
}
```

PowerShell's own mandatory-parameter validation already produces a
non-zero terminating error when a required parameter is missing and the
session is non-interactive — do not silently rely on the default
uncaught-error exit code (which is `1`, not `2`). Wrap the call in
`try`/`catch` at the entry point and remap that specific failure to `2`:

```powershell
try {
    Get-Greeting -Name $Name -ErrorAction Stop
    Exit-WithCode 0
} catch [System.Management.Automation.ParameterBindingException] {
    Write-Error $_
    Exit-WithCode 2
} catch {
    Write-Error $_
    Exit-WithCode 1
}
```

## 6. `--json` output and `--no-input`

Add a `-Json` switch parameter; serialize the same result object regardless
of branch, only the rendering differs:

```powershell
[Parameter()]
[switch]$Json
...
if ($Json) {
    $result | ConvertTo-Json -Depth 5
} else {
    Write-Output $result
}
```

Add a `-NoInput` switch (PowerShell convention: PascalCase,
`-NoInput`/`-Force`-style — not a GNU-style long flag) guarding every
interactive `Read-Host` call. When a required value is missing and
`-NoInput` is set, exit `2` rather than prompting:

```powershell
[Parameter()]
[switch]$NoInput

if (-not $Name) {
    if ($NoInput) {
        Write-Error "Name is required when -NoInput is set"
        Exit-WithCode 2
    }
    $Name = Read-Host "Enter name"
}
```

Never call `Read-Host` without first checking `-NoInput` — an unguarded
prompt hangs a non-interactive invocation (CI, piped input) instead of
failing fast. For cross-paradigm consistency with the other 11 languages
this skill targets, also accept the GNU-style spelling via parameter
aliasing rather than a second parameter:

```powershell
[Parameter()]
[Alias("no-input")]
[switch]$NoInput
```

## 7. stdout/stderr discipline

Route result data through the success/object output stream — plain
`return $result` or `Write-Output $result` — never through `Write-Host`.
This is PowerShell's *object* pipeline, not a text stream: a caller can
pipe the result into `ConvertTo-Json`, `Select-Object`, or another cmdlet
and get structured data back, not scraped text.

Route diagnostics to the error stream (`Write-Error`, stream 2) and
non-fatal notices to the warning stream (`Write-Warning`, stream 3):

```powershell
return $result              # stdout-equivalent: the object pipeline
Write-Error "fetch failed"  # stream 2: diagnostics
Write-Warning "using cached data"  # stream 3: non-fatal notice
```

Never use `Write-Host` for anything meant to be piped or composed —
`Write-Host` writes directly to the host UI and bypasses the output stream
entirely, so downstream cmdlets and `-Json` serialization never see it.
Reserve `Write-Host` only for interactive chrome explicitly excluded from
the pipeline (colored status banners under Section 4), never for data.

## 8. Distribution

Generate the manifest with `New-ModuleManifest`, not by hand-writing the
`.psd1`:

```powershell
New-ModuleManifest -Path .\MyTool\MyTool.psd1 `
    -RootModule "MyTool.psm1" `
    -ModuleVersion "0.1.0" `
    -FunctionsToExport @("Get-Greeting") `
    -Author "Your Name" `
    -Description "Short one-line description"
```

Publish to the PowerShell Gallery:

```powershell
Publish-Module -Path .\MyTool -NuGetApiKey $env:PSGALLERY_API_KEY
```

Document the end-user install command:

```powershell
Install-Module MyTool
```

## 9. Snapshot testing `--help`

Pester has no first-party snapshot-testing feature (unlike Verify.Xunit
for .NET) — use the honest golden-file pattern: capture `Get-Help` output,
diff it against a stored file with `Compare-Object`, and assert the diff
is empty.

```powershell
# Tests/MyTool.Tests.ps1
Describe "Get-Greeting help" {
    It "matches the golden help text" {
        $actual = Get-Help MyTool\Get-Greeting -Full | Out-String
        $golden = Get-Content "$PSScriptRoot\testdata\help.golden.txt" -Raw
        $diff = Compare-Object ($actual -split "`n") ($golden -split "`n")
        $diff | Should -BeNullOrEmpty
    }
}
```

Generate the initial golden file by capturing real output once and
reviewing it by hand before committing it:

```powershell
Get-Help MyTool\Get-Greeting -Full | Out-String | Set-Content Tests\testdata\help.golden.txt
```

Re-run that same capture command (without committing) whenever help text
changes intentionally, diff it manually, then overwrite the golden file —
never regenerate and commit blindly, since an unreviewed golden-file
update defeats the point of the snapshot.

## 10. Minimal worked example

Illustrative only — the actual generation is always freeform from this
reference, never a copy-paste of this exact code:

```powershell
# MyTool.psd1 (fields New-ModuleManifest writes; shown here for illustration)
@{
    RootModule        = 'MyTool.psm1'
    ModuleVersion      = '0.1.0'
    GUID               = 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
    Author             = 'Your Name'
    Description        = 'Greets people from the command line'
    FunctionsToExport  = @('Get-Greeting')
    PowerShellVersion  = '5.1'
}
```

```powershell
# Public/Get-Greeting.ps1
<#
.SYNOPSIS
    Builds and prints a greeting for the given name.
.PARAMETER Name
    The name to greet.
.PARAMETER Shout
    Emit the greeting in uppercase.
.PARAMETER Json
    Emit the result as JSON instead of plain text.
.EXAMPLE
    Get-Greeting -Name "World" -Shout
#>
function Get-Greeting {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory = $true, Position = 0)]
        [string]$Name,

        [Parameter()]
        [switch]$Shout,

        [Parameter()]
        [switch]$Json
    )
    $text = "Hello, $Name!"
    if ($Shout) { $text = $text.ToUpper() }
    $result = [pscustomobject]@{ Text = $text }

    if ($Json) { $result | ConvertTo-Json }
    else { return $result }
}
```
