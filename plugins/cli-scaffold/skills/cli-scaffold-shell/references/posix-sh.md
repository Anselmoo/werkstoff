# POSIX sh CLI generation reference

Pin every POSIX sh CLI generated by this skill to the idioms below. POSIX sh
here means the portable lowest-common-denominator dialect that runs
correctly and identically under **dash**, **ksh**, and **bash invoked as
`/bin/sh`** â€” targeted at minimal server/container environments (Alpine/
BusyBox, Debian's `/bin/sh` being dash) where a full bash is not guaranteed
to be installed. This dialect is **stricter** than this skill's `bash.md`
and `zsh.md` references â€” do not borrow anything bash- or zsh-specific from
them. Regenerate the same structure every run; do not invent alternative
shapes.

## 1. Argument parsing

Use POSIX `getopts` only â€” the one parsing tool genuinely common to dash,
ksh, and bash. Never use GNU `getopt(1)` (not portable, not present on
BusyBox) and never hand-roll a `while [ "$#" -gt 0 ]` positional-shift
parser for flags that take short options.

Loop shape â€” every construct inside must itself be POSIX-compliant (`[ ]`,
not `[[ ]]`; `name() { }`, not `function name`):

```sh
verbose=0
output=""

while getopts ":o:vh" opt; do
  case "$opt" in
    o) output=$OPTARG ;;
    v) verbose=1 ;;
    h) usage; exit 0 ;;
    :) printf 'error: option -%s requires an argument\n' "$OPTARG" >&2; exit 2 ;;
    \?) printf 'error: unknown option -%s\n' "$OPTARG" >&2; exit 2 ;;
  esac
done
shift $((OPTIND - 1))
```

Leading `:` in the optstring puts `getopts` into silent-error mode so the
script controls the `:`/`\?` messages itself, instead of relying on each
shell's differently worded built-in error text. `getopts` only supports
single-character short options (`-o`, `-v`) â€” there is no POSIX long-option
(`--output`) mechanism; document that limitation rather than faking long
options with a manual pre-parser.

## 2. Forbidden bashisms

Each of these breaks portability to dash/BusyBox. Use the POSIX replacement
instead, with no exceptions inside `lib/<app>.sh` or `bin/<app>`.

| Bashism | POSIX replacement |
|---|---|
| Arrays: `arr=(a b c)`, `${arr[@]}` | No POSIX array type. Use a newline- or space-delimited string with a `for` loop, or one variable per item, or `set --` to load positional params: `set -- a b c; for x in "$@"; do ...; done` |
| `[[ "$x" == foo ]]` | `[ "$x" = foo ]` (POSIX `test`; note `=`, not `==`) |
| `local var=val` inside a function | Not POSIX. See caveat below. |
| `function name() { ... }` | `name() { ... }` (the only POSIX form) |
| `[ "$x" == "$y" ]` | `[ "$x" = "$y" ]` (`==` is a bash/ksh extension to `test`; POSIX `test` only defines `=`) |
| Here-strings: `cmd <<< "$x"` | `printf '%s\n' "$x" \| cmd` |
| `$(( x ** 2 ))`, `$(( x++ ))`, and other non-basic `$(( ))` extensions | Basic POSIX arithmetic only: `+ - * / %`, comparison, and assignment inside `$(( ))`; for anything fancier shell out to `awk` |
| Process substitution: `diff <(cmd1) <(cmd2)` | Write to temp files and diff those, or pipe one side: `cmd1 \| diff - <(cmd2)` is itself non-POSIX â€” use `cmd1 > /tmp/a; cmd2 > /tmp/b; diff /tmp/a /tmp/b` with `trap ... EXIT` cleanup |

`local` caveat: strict POSIX does not define `local`. In practice it is
supported by dash, ksh, bash, and zsh alike, making it a de facto universal
extension â€” the pragmatic default for this skill's generated scripts is to
**use `local`** for function-scoped variables. Only drop it (fall back to a
subshell â€” `result=$(compute_thing)` â€” or a careful unique-prefixed global
variable naming convention, e.g. `_computefoo_tmp`) when the user explicitly
asks for true strict-POSIX compliance with no de facto extensions at all.


`lib/myapp.sh` defines every function with the POSIX `name() { ... }` form
only, and does nothing at source time beyond defining functions (no top-level
side effects â€” no output, no `exit`, no environment mutation on `. lib/myapp.sh`).
`bin/myapp` is `#!/bin/sh` with POSIX `set -eu` at the top, sources
`lib/myapp.sh`, then runs the `getopts` loop and dispatches to functions.
Note `pipefail` is **not POSIX** â€” dash and BusyBox `ash` do not support it â€”
so omit it; if a pipeline's exit status matters, check each stage's status
explicitly or restructure to avoid the pipe.

## 4. Help text and completions

Hand-write a `usage()` function in `lib/myapp.sh`, called from `bin/myapp`
on `-h` or on a `getopts` parse error:

```sh
usage() {
  cat <<'EOF'
Usage: myapp [-v] [-o FILE] <command>

  -o FILE   write output to FILE
  -v        verbose output
  -h        show this help
EOF
}
```

POSIX sh has **no native completion-registration mechanism** â€” completions
are a feature of the *interactive* shell (bash's `complete`/`compgen`, zsh's
`compsys`), not of the POSIX sh language or of a script run via `#!/bin/sh`.
If a POSIX sh script gets sourced by an interactive bash or zsh session, any
completions it wants come from that session's own completion system â€”
documented in `bash.md`/`zsh.md`, not something `sh` itself can register.
State this limitation plainly in generated docs rather than fabricating a
completion script that a POSIX-mode shell cannot actually load.

## 5. NO_COLOR-aware output

```sh
use_color=1
[ -n "${NO_COLOR:-}" ] && use_color=0
[ -t 1 ] || use_color=0
```

Both tests are POSIX-safe: `[ -n "${NO_COLOR:-}" ]` uses POSIX `[ ]` (never
`[[ ]]`) with `${var:-}` default-expansion under `set -u`, and `[ -t 1 ]` is
a portable `test` primitive for "is stdout a terminal" across dash/ksh/bash.
Gate every ANSI escape emission behind `use_color`; never emit raw escape
codes unconditionally.

## 6. Exit codes

Frozen cross-paradigm contract, identical to `bash.md`, expressed in strict
POSIX syntax:

- `0` â€” success
- `1` â€” general/runtime error
- `2` â€” usage/argument error

```sh
while getopts ":o:vh" opt; do
  case "$opt" in
    o) output=$OPTARG ;;
    v) verbose=1 ;;
    h) usage; exit 0 ;;
    :) printf 'error: -%s requires an argument\n' "$OPTARG" >&2; exit 2 ;;
    \?) printf 'error: unknown option -%s\n' "$OPTARG" >&2; exit 2 ;;
  esac
done

do_work() {
  some_command || { printf 'error: some_command failed\n' >&2; return 1; }
}

do_work || exit 1
```

## 7. `--json` output and `--no-input`

Do not assume `jq` is installed â€” BusyBox/minimal images this dialect
targets often omit it. Emit JSON by hand with `printf`, treating `jq` as
optional-if-available for anything more than trivial flat objects:

```sh
if [ "$json_output" -eq 1 ]; then
  printf '{"status":"%s","count":%d}\n' "$status" "$count"
else
  printf 'status: %s (count=%d)\n' "$status" "$count"
fi
```

`--no-input` guards every interactive `read`. There is no `read -p` in
POSIX â€” that flag is a bashism â€” so prompt with `printf` on its own line,
then `read` on the next:

```sh
if [ -z "$name" ]; then
  if [ "$no_input" -eq 1 ]; then
    printf 'error: --name is required with --no-input\n' >&2
    exit 2
  fi
  printf 'Project name: '
  read -r name
fi
```

## 8. stdout/stderr discipline

Prefer `printf` over `echo` for anything beyond a literal fixed string â€”
`echo`'s handling of backslash escapes and flags (`-e`, `-n`) is
implementation-defined and differs across dash/ksh/bash-as-sh, a real
portability trap. Route results (data a downstream pipe consumes) to
stdout; route diagnostics, progress, and logs to stderr:

```sh
printf '%s\n' "$result"                 # stdout: data
printf 'fetching...\n' >&2              # stderr: progress/diagnostic
```

Never let a diagnostic line land on stdout â€” it corrupts `--json` output
and breaks piping into `jq`/`grep`/downstream tools.

## 9. Distribution

Same channel as `bash.md`'s Homebrew/apt section â€” cross-reference it
rather than repeating package-manager mechanics here. POSIX sh's most
common real-world distribution shape, though, is different: a single
self-contained `#!/bin/sh` file fetched and executed directly, e.g.
`curl -fsSL https://example.com/myapp | sh`, for minimal/container
environments where installing via a package manager isn't practical. When
generating for that shape, keep `bin/myapp` genuinely self-contained (inline
`lib/myapp.sh`'s functions rather than requiring a separate `source` step)
so the curl-pipe-sh invocation works without a companion file.

## 10. Snapshot testing `--help`

Use **shunit2**, not bats-core â€” bats-core itself requires bash to run,
which defeats the purpose of testing a script meant to also run correctly
under dash/BusyBox with no bash present. shunit2 is a POSIX-compatible
shell test framework runnable under `sh` itself.

```sh
# test/myapp_test.sh
testHelpMatchesGolden() {
  actual=$(../bin/myapp -h)
  expected=$(cat testdata/help.golden)
  assertEquals "$expected" "$actual"
}

# shunit2 must be sourced last
. shunit2
```

Run with `sh test/myapp_test.sh` (or under dash explicitly:
`dash test/myapp_test.sh`) so the test itself proves POSIX compliance, not
just the script under test. Regenerate `testdata/help.golden` by capturing
`bin/myapp -h` output whenever `usage()` changes intentionally, and diff it
in review rather than hand-editing the golden file.

## 11. Minimal worked example

Illustrative only â€” the actual generation is always freeform from this
reference, never a copy-paste of this exact code.

```sh
# lib/myapp.sh â€” sourced function library, zero side effects at source-time
usage() {
  printf 'Usage: myapp [-n NAME] [-j] [-h]\n'
}

greet() {
  name=$1
  shout=$2
  if [ -z "$name" ]; then
    printf 'error: name must not be empty\n' >&2
    return 1
  fi
  if [ "$shout" -eq 1 ]; then
    printf '%s\n' "$name" | tr '[:lower:]' '[:upper:]'
  else
    printf 'Hello, %s!\n' "$name"
  fi
}
```

```sh
#!/bin/sh
# bin/myapp â€” thin dispatcher
set -eu

script_dir=$(CDPATH= cd -- "$(dirname -- "$0")/../lib" && pwd)
. "$script_dir/myapp.sh"

name=""
shout=0
json=0

while getopts ":n:jh" opt; do
  case "$opt" in
    n) name=$OPTARG ;;
    j) json=1 ;;
    h) usage; exit 0 ;;
    :) printf 'error: -%s requires an argument\n' "$OPTARG" >&2; exit 2 ;;
    \?) printf 'error: unknown option -%s\n' "$OPTARG" >&2; exit 2 ;;
  esac
done

if [ "$json" -eq 1 ]; then
  printf '{"name":"%s"}\n' "$name"
else
  greet "$name" "$shout" || exit 1
fi
```
