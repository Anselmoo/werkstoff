# Zsh CLI generation reference

Pin every Zsh CLI generated by this skill to the idioms below. There is no
stored template — regenerate freeform from this reference every time, but
converge on the same shape every time. Do not invent alternative structures.

## 1. Argument parsing

Use **`zparseopts`**, Zsh's own builtin, never `getopts`. This is a
Zsh-specific choice, not a general shell-scripting default: Bash's `getopts`
only understands single-character options and requires a hand-rolled
`case "$1"`/`shift` loop bolted on top to support `--long` flags at all.
`zparseopts` understands `--long` options natively, as first-class spec
entries — that native long-option support is *the* reason to prefer
`zparseopts` over `getopts` specifically when the target is Zsh. Do not use
`getopts` in a Zsh script generated by this skill under any circumstance.

Pinned idiom:

```zsh
zparseopts -D -E -F -- \
  h=help o:=output j=json_flag \
  -help=help -json=json_flag \
  -output:=output || { usage >&2; exit 2 }
```

Read the flags literally, every generation:

- `-D` — strip matched options out of the positional parameter list (`$@`)
  once parsed, so remaining args are pure positional operands.
- `-E` — keep parsing past the first non-option argument instead of
  stopping there (needed once operands are interleaved with flags).
- `-F` — fail (non-zero `$?`) on any unrecognized option instead of
  silently ignoring it; without `-F`, `zparseopts` swallows unknown flags,
  which defeats usage validation. Always include `-F`.
- `x=arr` — boolean flag `-x`; presence appends into array `arr`.
- `x:=arr` — flag `-x` that consumes one argument; both the flag and its
  value land in `arr` (read the value via `${arr[-1]}`).
- `-help=help` / `-json=json_flag` merge the long form into the *same*
  destination array as its short form, so one presence check
  (`(( ${#help} ))`) covers both spellings.

`zparseopts` never exits on its own — it only sets `$?`. Checking that
`$?` explicitly (as the `|| { usage >&2; exit 2 }` above does) is mandatory;
see §5.

## 2. Project layout

Generate exactly this tree:

```
<app>/
├── bin/
│   └── <app>                # thin executable entry point
├── lib/
│   └── <app>.zsh             # sourced function library, no side effects at source-time
├── completions/
│   └── _<app>                # zsh compdef completion function
├── Formula/
│   └── <app>.rb               # Homebrew formula
├── packaging/
│   └── deb/                   # optional apt/.deb control files (fallback, not default)
├── test/
│   ├── <app>.bats             # bats-core specs, incl. --help golden-file diff
│   └── testdata/
│       └── help.golden         # captured `bin/<app> --help` output
└── README.md
```

`lib/<app>.zsh` rule (non-negotiable): only `function name { ... }`
definitions at the top level — no top-level variable assignment, no code
that runs on `source`. It must be safe to `source lib/<app>.zsh` from a bats
test in complete isolation and call individual functions without ever
invoking `bin/<app>`.

`bin/<app>` rule: resolve its own directory with `${0:A:h}` (never a
hardcoded or `pwd`-relative path) so `source`-ing `lib/<app>.zsh` works
regardless of the caller's cwd or symlinks:

```zsh
#!/usr/bin/env zsh
emulate -L zsh
setopt err_exit no_unset pipe_fail

source "${0:A:h}/../lib/<app>.zsh"
```

## 3. Help text and completions

Hand-write a `usage()` function — the same shape as the Bash reference's, so
both paradigms present identical `--help` text for the same tool:

```zsh
usage() {
  cat <<EOF
Usage: <app> [-h|--help] [-o|--output FILE] [-j|--json] NAME

Options:
  -h, --help          Show this help and exit
  -o, --output FILE   Write result to FILE instead of stdout
  -j, --json          Emit JSON instead of plain text
EOF
}
```

Call it on `-h`/`--help` (exit 0) and on any usage error (to stderr, exit 2).

For completions, generate `completions/_<app>` as a native Zsh completion
function — this is Zsh's real advantage over Bash here, since Bash's
`complete`/`compgen` only offers word-list/function callbacks, while Zsh's
`_arguments` builtin gives structured, self-documenting option specs with
type-aware value completion. Minimal idiom:

```zsh
#compdef <app>

_arguments \
  '(-h --help)'{-h,--help}'[show help and exit]' \
  '(-o --output)'{-o,--output}'[write result to FILE]:file:_files' \
  '(-j --json)'{-j,--json}'[emit JSON]' \
  '1:name:_default'
```

Install `_<app>` into a directory on `$fpath` (Homebrew's formula does this
automatically via `zsh_completion.install "completions/_<app>"`; for manual
installs, document copying it into `/usr/local/share/zsh/site-functions`).

## 4. NO_COLOR-aware output

Zsh has no built-in `NO_COLOR` support — check it explicitly, combined with
a TTY check, exactly like the Bash reference:

```zsh
[[ -n "${NO_COLOR:-}" ]] && USE_COLOR=0 || USE_COLOR=1
[[ -t 1 ]] || USE_COLOR=0
```

Compute this once near the top of `bin/<app>` (after sourcing `lib/<app>.zsh`)
and gate every colored `print` call on `$USE_COLOR`. Never emit raw ANSI
escapes from inside `lib/<app>.zsh` — keep color decisions in `bin/<app>`.

## 5. Exit codes

Frozen cross-paradigm contract, identical across every language this plugin
generates:

- `0` — success
- `1` — general/runtime error
- `2` — usage/argument error

`zparseopts` sets `$?` on failure (unknown option with `-F`, missing
argument to a `x:` spec) but never exits by itself — check it explicitly and
exit 2:

```zsh
zparseopts -D -E -F h=help j=json_flag -help=help -json=json_flag \
  || { usage >&2; exit 2 }
```

Map runtime failures from sourced `lib/<app>.zsh` functions to exit 1 at the
call site in `bin/<app>`:

```zsh
result=$(app::run "$name") || { print -u2 "error: run failed"; exit 1 }
```

## 6. `--json` output and `--no-input`

Capture `--json` via the same `zparseopts` call as §1 (`j=json_flag
-json=json_flag`); branch only in `bin/<app>`, never inside
`lib/<app>.zsh` functions, which must return plain values:

```zsh
if (( ${#json_flag} )); then
  print -r -- "{\"name\":\"${name}\"}"
else
  print -r -- "Hello, ${name}!"
fi
```

Guard every interactive `read` with a `--no-input` flag captured the same
way (`n=no_input -no-input=no_input`). If a required value is missing and
`--no-input` is set, fail fast at exit 2 instead of blocking on `read`:

```zsh
if [[ -z "$name" ]]; then
  if (( ${#no_input} )); then
    print -u2 "error: NAME is required with --no-input"
    exit 2
  fi
  read -r "name?Name: "
fi
```

## 7. stdout/stderr discipline

Use `print` for stdout results, not `echo` — `print`'s `-r` (raw, no escape
processing) and `-n`/`-u<fd>` flags give exact control `echo` lacks, and this
matters wherever output is piped into `jq`/`diff`/downstream tooling:

```zsh
print -r -- "$result"      # stdout: exact data, no interpretation
```

For diagnostics and logs, use `print -u2` (Zsh's file-descriptor-targeted
`print`) or plain `echo "..." >&2` — either is acceptable, but be consistent
within one generated CLI:

```zsh
print -u2 -- "fetching ${name}..."
```

Never let a diagnostic line reach stdout — it corrupts `--json` output and
breaks piping.

## 8. Distribution

Ship via **Homebrew** as the default path — one `Formula/<app>.rb`:

```ruby
class <App> < Formula
  desc "..."
  url "https://example.com/<app>-1.0.0.tar.gz"
  sha256 "..."

  def install
    bin.install "bin/<app>"
    prefix.install "lib"
    zsh_completion.install "completions/_<app>"
  end
end
```

```bash
brew install --build-from-source Formula/<app>.rb
```

Document **apt** only as a lighter-weight *optional* fallback, not a
default — packaging for apt means hand-building a `.deb` control tree under
`packaging/deb/`, which is meaningfully heavier-weight than a single Ruby
formula file. Only generate `packaging/deb/` when the user explicitly asks
for `.deb` distribution:

```bash
dpkg-deb --build packaging/deb <app>_1.0.0_all.deb
sudo apt install ./<app>_1.0.0_all.deb
```

## 9. Snapshot testing `--help`

Use **bats-core** with a golden-file diff. bats itself runs on Bash, but
`run` simply execs the target script through its own shebang — since
`bin/<app>` starts with `#!/usr/bin/env zsh`, bats runs it as Zsh
transparently, no special bats configuration needed:

```bats
#!/usr/bin/env bats

@test "--help matches golden file" {
  run "${BATS_TEST_DIRNAME}/../bin/<app>" --help
  [ "$status" -eq 0 ]
  diff <(printf '%s\n' "$output") "${BATS_TEST_DIRNAME}/testdata/help.golden"
}
```

Regenerate the golden file deliberately, never by hand-editing it:

```bash
bin/<app> --help > test/testdata/help.golden
```

Run bats without regenerating in CI, so any unreviewed `--help` drift fails
the build:

```bash
bats test/<app>.bats
```

## 10. Minimal worked example

Illustrative only — regenerate concrete names, functions, and error
handling to fit the actual CLI being built; do not copy this verbatim.

```zsh
# lib/greet.zsh — sourced function library, zero side effects at source-time
function greet::build_message {
  local name="$1" shout="$2"
  [[ -z "$name" ]] && { print -u2 "error: name must not be empty"; return 1 }
  local msg="Hello, ${name}!"
  (( shout )) && msg="${msg:u}"
  print -r -- "$msg"
}
```

```zsh
#!/usr/bin/env zsh
# bin/greet — thin executable entry point
emulate -L zsh
setopt err_exit no_unset pipe_fail

source "${0:A:h}/../lib/greet.zsh"

usage() { cat <<EOF
Usage: greet [-h] [-j] NAME
EOF
}

zparseopts -D -E -F h=help j=json_flag -help=help -json=json_flag \
  || { usage >&2; exit 2 }
(( ${#help} )) && { usage; exit 0 }

name="${1:-}"
[[ -z "$name" ]] && { print -u2 "error: NAME is required"; exit 2 }

msg=$(greet::build_message "$name" 0) || exit 1
if (( ${#json_flag} )); then
  print -r -- "{\"text\":\"${msg}\"}"
else
  print -r -- "$msg"
fi
```
