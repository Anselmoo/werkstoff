# Python CLI generation reference

Pin every Python CLI generated by this skill to the idioms below. Do not
invent alternative shapes — regenerate the same structure every run.

## 1. Framework

Default to **Typer**. Only fall back to **argparse** (stdlib only) when the
user explicitly requests zero external dependencies ("no dependencies",
"stdlib only", "no pip installs"). Never blend the two paths in one
generation — pick one and use it consistently across `cli.py`, `pyproject.toml`,
and the worked example.

Decision rule:

```
user said "no dependencies" / "stdlib only"?
  yes -> argparse
  no  -> Typer (default)
```

Install (Typer path):

```bash
uv add typer
# or
pip install typer
```

Minimal invocation shape (Typer):

```python
import typer

app = typer.Typer(no_args_is_help=True)

@app.command()
def main(name: str) -> None:
    ...

if __name__ == "__main__":
    app()
```

Minimal invocation shape (argparse, stdlib-only path):

```python
import argparse

def build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(prog="mytool")
    parser.add_argument("name")
    return parser

def main() -> int:
    args = build_parser().parse_args()
    ...
    return 0

if __name__ == "__main__":
    raise SystemExit(main())
```

## 2. Project layout

Generate this exact tree (single-file `core.py` for small tools; expand to a
`core/` package only once business logic outgrows one file):

```
mypkg/
├── pyproject.toml
├── README.md
├── src/
│   └── mypkg/
│       ├── __init__.py
│       ├── core.py        # business logic, zero CLI imports
│       └── cli.py         # Typer/argparse entry point, thin
└── tests/
    ├── __init__.py
    ├── test_core.py
    ├── test_cli.py
    └── __snapshots__/      # syrupy snapshot dir, generated on first run
```

For larger tools, replace `core.py` with a `core/` package (`core/__init__.py`,
`core/<domain>.py`, ...) but keep the same rule: nothing under `core/` imports
`typer`, `argparse`, `click`, or `rich`.

`core.py` rule (non-negotiable): every function/class in it must import and
run cleanly from a plain Python REPL with no CLI framework loaded. If a
function needs `typer.Exit` or `argparse.Namespace` to make sense, it belongs
in `cli.py`, not `core.py`.

`cli.py` rule: only argument parsing, calling into `core.py`, output
formatting (text vs `--json`), and exit-code mapping. No conditionals that
encode business rules — those live in `core.py` so they're testable without
spinning up a CLI runner.

## 3. Help text and completions

Typer auto-generates `--help` from type hints, docstrings, and `typer.Option`/
`typer.Argument` metadata. Structurally it always produces:

```
Usage: mytool [OPTIONS] COMMAND [ARGS]...

  <docstring of the app or command>

Options:
  --install-completion  Install completion for the current shell.
  --show-completion     Show completion for the current shell.
  --help                 Show this message and exit.

Commands:
  <subcommand>  <first line of its docstring>
```

For a single-command (non-group) app, the `Commands:` section is replaced by
an `Arguments:`/`Options:` pair listing the command's own parameters. Rely on
this auto-generated structure — never hand-write a `--help` string.

Shell completions ship built in; do not hand-roll a completion script.
Generate/install with:

```bash
mytool --install-completion   # detects shell, writes the completion script
mytool --show-completion      # prints the script instead of installing it
```

For the argparse fallback path, state explicitly that shell completions are
out of scope unless the user adds `argcomplete` as a dependency — do not
silently promise Typer-level completion behavior from argparse.

## 4. NO_COLOR-aware output

Typer's `--help` and any `rich.console.Console` output already honor the
`NO_COLOR` environment variable automatically — Rich detects it at console
construction time and disables ANSI styling. Do not add custom `NO_COLOR`
plumbing for the default color path.

When the generated CLI creates its own `Console` for styled output (tables,
progress, colored diagnostics), pass color control explicitly rather than
relying on autodetection alone, so behavior is verifiable in tests:

```python
import os
from rich.console import Console

console = Console(no_color=bool(os.environ.get("NO_COLOR")))
```

To verify NO_COLOR is honored, run the built CLI with `NO_COLOR=1 mytool ...`
and assert the captured output contains no ANSI escape sequences.

## 5. Exit codes

Frozen cross-paradigm contract — every generated CLI, regardless of language,
must honor this:

- `0` — success
- `1` — general/runtime error (caught exception, failed operation)
- `2` — usage/argument error (bad flags, missing required argument)

Typer/Click already auto-emit exit code `2` on argument-parsing failures
(unknown option, missing required argument, wrong type) — do not add manual
handling for that case. Application code only needs to cover `0` and `1`
explicitly:

```python
import typer

@app.command()
def run() -> None:
    try:
        do_work()
    except SomeExpectedError as exc:
        typer.echo(f"error: {exc}", err=True)
        raise typer.Exit(code=1)
    # implicit typer.Exit(code=0) on normal return
```

For a case that is a usage error detected inside application logic (e.g. two
mutually exclusive flags both set, valid at parse time but invalid together),
raise `typer.Exit(code=2)` explicitly rather than `code=1` — it is still a
usage error, just one Click's parser couldn't catch on its own.

argparse fallback: return/`raise SystemExit(0|1)` from `main()`; argparse
itself calls `parser.error(...)` internally and exits with `2` on parse
failures, so again only `0`/`1` need explicit code.

## 6. `--json` output and `--no-input`

`--json` flag: keep exactly one rendering branch difference in `cli.py` —
call the same `core.py` function regardless, then switch how the result is
printed:

```python
import json
import typer

@app.command()
def status(json_output: bool = typer.Option(False, "--json")) -> None:
    result = core.get_status()
    if json_output:
        typer.echo(json.dumps(result))
    else:
        typer.echo(f"status: {result.state}")
```

`--no-input` flag: disable every interactive prompt in that mode instead of
silently defaulting. If a required value is missing and `--no-input` is set,
error out at exit code `2` (usage error — the invocation was incomplete for
non-interactive use):

```python
@app.command()
def init(
    name: str = typer.Option(None, "--name"),
    no_input: bool = typer.Option(False, "--no-input"),
) -> None:
    if name is None:
        if no_input:
            typer.echo("error: --name is required with --no-input", err=True)
            raise typer.Exit(code=2)
        name = typer.prompt("Project name")
    ...
```

Never call `typer.prompt()` or `typer.confirm()` without first checking
`no_input` — an unguarded prompt call hangs a non-interactive invocation
(CI, piped stdin) instead of failing fast.

## 7. stdout/stderr discipline

Route results and data — the thing a downstream `|` pipe consumes — to
stdout via `typer.echo()` (or plain `print()`). Route diagnostics, progress,
and log lines to stderr:

```python
typer.echo(result)                    # stdout: data
typer.echo("fetching...", err=True)   # stderr: progress/diagnostic
```

For anything beyond a one-line diagnostic, configure Python `logging` to
stream to stderr instead of stacking `err=True` calls:

```python
import logging
import sys

logging.basicConfig(stream=sys.stderr, level=logging.INFO, format="%(message)s")
logger = logging.getLogger("mytool")
```

Never let a log call or progress message land on stdout — it corrupts
`--json` output and breaks piping into `jq`/`grep`/downstream tools.

## 8. Distribution

Generate a `pyproject.toml` with a `[project.scripts]` entry point:

```toml
[project]
name = "mytool"
version = "0.1.0"
dependencies = ["typer>=0.12"]

[project.scripts]
mytool = "mytool.cli:app"

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```

(For the argparse stdlib-only path, drop the `dependencies` line entirely and
point the entry point at `"mytool.cli:main"`.)

Document both supported local-install commands:

```bash
uv tool install .
# or
pipx install .
```

Document publishing to PyPI with either toolchain:

```bash
uv build
uv publish
# or
uv build
twine upload dist/*
```

## 9. Snapshot testing `--help`

Use pytest + syrupy to pin `--help` output. Generate a `tests/test_cli.py`
that invokes the Typer app through `CliRunner` and asserts the captured
stdout against a syrupy snapshot:

```python
from typer.testing import CliRunner
from mypkg.cli import app

# Rich/Typer's --help rendering wraps to the terminal width (flaky across
# machines/CI unless pinned) and, separately, Click's rich_utils forces
# ANSI color whenever GITHUB_ACTIONS is set — which is unconditionally
# true on every GitHub Actions runner — even into this non-tty captured
# buffer, and that forcing wins over NO_COLOR in this code path. Setting
# NO_COLOR=1 alone is not sufficient; the CI-detection vars must be
# explicitly cleared too.
runner = CliRunner(
    env={
        "COLUMNS": "80",
        "NO_COLOR": "1",
        "GITHUB_ACTIONS": "",
        "FORCE_COLOR": "",
        "CLICOLOR_FORCE": "",
    }
)

def test_help_snapshot(snapshot) -> None:
    result = runner.invoke(app, ["--help"])
    assert result.exit_code == 0
    assert result.output == snapshot
```

Never construct `CliRunner()` with no `env=` override for a `--help`
snapshot test — two independent sources of flakiness, both confirmed by
reproducing an actual GitHub Actions failure end-to-end (real pty +
`GITHUB_ACTIONS=true`, matching the runner's own default environment):

- The recorded snapshot only matches whatever terminal width happened to
  be active when it was generated, and fails non-deterministically
  (`1 failed` locally, or in CI on a runner with a different default
  width) for anyone else — pin `COLUMNS`. Any fixed value works as long
  as it's stable.
- Separately, and independently of width: when the *invoking* process
  (e.g. `uv run pytest` in a CI step) has a real pty attached — which
  GitHub Actions steps do, for live log rendering — Click's `rich_utils`
  detects `GITHUB_ACTIONS` in the environment and forces ANSI codes into
  the help output, even though `CliRunner` itself captures to a plain,
  non-tty `StringIO`. This happens regardless of the actual sub-buffer's
  tty-ness, and it is not fixed by `NO_COLOR=1` alone — `GITHUB_ACTIONS`,
  `FORCE_COLOR`, and `CLICOLOR_FORCE` must all be explicitly overridden
  to empty in the same `env=` dict. A snapshot recorded locally (no pty,
  no `GITHUB_ACTIONS`) will otherwise pass locally and fail in CI only,
  with a diff that looks like every line changed but reads as identical
  text — the invisible difference is embedded ANSI escape sequences.

Add `syrupy` as a dev dependency (the PyPI/`uv add` package name is `syrupy`,
not `pytest-syrupy` — the latter does not exist as a distribution) and
generate/update the stored snapshot with:

```bash
uv run pytest --snapshot-update
```

Run the same command without `--snapshot-update` in CI so any unreviewed
`--help` drift fails the build instead of silently passing.

## 10. Minimal worked example

Illustrative only — the actual generation is always freeform from this
reference, never a copy-paste of this exact code:

```python
# src/mypkg/core.py — no CLI imports, REPL-importable
from dataclasses import dataclass

@dataclass
class Greeting:
    text: str

def build_greeting(name: str, shout: bool = False) -> Greeting:
    if not name:
        raise ValueError("name must not be empty")
    text = f"Hello, {name}!"
    return Greeting(text.upper() if shout else text)
```

```python
# src/mypkg/cli.py — thin Typer entry point
import json
import typer
from mypkg import core

app = typer.Typer(no_args_is_help=True)

@app.command()
def greet(
    name: str,
    shout: bool = typer.Option(False, "--shout"),
    json_output: bool = typer.Option(False, "--json"),
) -> None:
    try:
        result = core.build_greeting(name, shout=shout)
    except ValueError as exc:
        typer.echo(f"error: {exc}", err=True)
        raise typer.Exit(code=1)
    if json_output:
        typer.echo(json.dumps({"text": result.text}))
    else:
        typer.echo(result.text)

if __name__ == "__main__":
    app()
```
