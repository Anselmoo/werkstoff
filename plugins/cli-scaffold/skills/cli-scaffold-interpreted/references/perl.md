# Perl CLI generation reference

Pin every Perl CLI generated by this skill to the idioms below. Do not invent
alternative shapes — regenerate the same structure every run.

## 1. Framework

Use **`Getopt::Long`** for argument parsing — it is core Perl (ships with
every Perl install since 5.x), so it needs no dependency declaration beyond
a `use` statement. Never reach for `Getopt::Std`, `Getopt::Long::Descriptive`,
or a hand-rolled `@ARGV` walker — `Getopt::Long` is the near-universal
standard and is what the rest of this reference assumes.

Scaffold the distribution itself with **Minilla**, the modern lightweight
tool purpose-built for minimal CPAN-distributable tools:

```bash
cpanm --with-recommends Minilla
minil new App-MyTool
cd App-MyTool/
```

Minimal `GetOptions(...)` shape — always destructure into a plain hash, never
into loose lexicals, so the parsed-options bundle can be passed to `Core.pm`
as a single argument:

```perl
use Getopt::Long qw(GetOptions);

my %opt = (shout => 0, json => 0, no_input => 0);
GetOptions(
    \%opt,
    'name=s',
    'shout',
    'json',
    'no_input|no-input',
    'help',
) or exit(2);
```

`Getopt::Long` stores a parsed option under a hash key equal to the
**first** name in its spec, not an underscore-normalized form of
whatever flag was typed — a bare `'no-input'` spec stores into
`$opt{'no-input'}` (hyphen), not `$opt{no_input}`. Since every other
idiom in this file (and generated code) reads `$opt{no_input}`, the spec
must list `no_input` first and `no-input` as an alias, exactly as shown
above, so the CLI-facing flag stays `--no-input` (GNU-style, matching the
doctrine) while the hash key stays a valid, underscore-only Perl
identifier.

## 2. Project layout

Generate this exact tree — it is Minilla's actual `minil new` output plus the
one manually-added executable:

```
App-MyTool/
├── cpanfile
├── Changes
├── README.md
├── minil.toml
├── lib/
│   └── App/
│       └── MyTool.pm       # thin package marker; delegate to Core.pm
│   └── App/
│       └── MyTool/
│           └── Core.pm     # business logic, zero Getopt::Long/CLI code
├── script/
│   └── mytool              # thin Getopt::Long entry point, executable bit set
└── t/
    ├── 00_compile.t
    ├── core.t
    └── data/
        └── help.txt         # golden file for --help snapshot testing
```

`script/` is Minilla's default executable directory — its `script_files`
config defaults to `['script/*', 'bin/*']` (script checked first). `minil new`
does not scaffold an executable automatically (it assumes a module-only
dist by default); add `script/<app>` yourself and mark it executable
(`chmod +x script/mytool`).

`lib/App/MyTool/Core.pm` rule (non-negotiable): every sub in it must be
callable from `perl -MApp::MyTool::Core -e '...'` with no CLI framework
loaded — no `Getopt::Long`, no `exit()`, no reads from `@ARGV` or `STDIN`
inside `Core.pm` itself. Return values and `die`/exceptions only; let the
caller decide exit codes.

`script/mytool` rule: only argument parsing, calling into `Core.pm`, output
formatting (text vs `--json`), and exit-code mapping. No business-rule
conditionals — those live in `Core.pm` so they're testable without invoking
the script.

## 3. Help text and completions

`Getopt::Long` has no auto-generated help — write a `sub usage` by hand and
call it both for `--help` and for any missing-required-argument failure, so
the two paths never drift out of sync:

```perl
sub usage {
    print <<"USAGE";
Usage: mytool [OPTIONS] NAME

Options:
  --shout           Uppercase the greeting
  --json            Emit machine-readable JSON
  --no-input        Fail instead of prompting for missing values
  --help            Show this message and exit
USAGE
    return;
}

GetOptions(\%opt, 'name=s', 'shout', 'json', 'no_input|no-input', 'help')
    or do { usage(); exit(2); };

if ($opt{help}) {
    usage();
    exit(0);
}
```

Perl has no idiomatic first-party shell-completion generator — do not invent
one or claim `Getopt::Long` produces completions. The honest fallback is a
hand-maintained completion script shipped alongside the dist:
`completions/mytool.bash`, referenced from `README.md`, installed manually by
the user (`source completions/mytool.bash` or copied into their
`bash-completion` directory). State this limitation explicitly rather than
promising Typer/Click-level auto-completion.

## 4. NO_COLOR-aware output

`Term::ANSIColor` (core since Perl 5.x, and specifically since its own
5.01 release in 2020) already honors `NO_COLOR` automatically — when
`$ENV{NO_COLOR}` is set (to any value), `colored()` returns its text
unstyled with no ANSI codes, with no application-code check required.
Do not add a hand-rolled `NO_COLOR` wrapper around `colored()` calls on
the assumption that Perl needs one; that would be redundant, not wrong.

```perl
use Term::ANSIColor qw(colored);

print colored("done", 'green'), "\n";
```

If the generated CLI builds its own ANSI escapes directly instead of
going through `Term::ANSIColor` (e.g. hand-written `\e[32m` sequences),
`NO_COLOR` awareness is not automatic for that path — gate those specific
call sites on `$ENV{NO_COLOR}` explicitly, same as any other
hand-rolled-escape language in this reference set.

## 5. Exit codes

Frozen cross-paradigm contract — every generated CLI, regardless of language,
must honor this:

- `0` — success
- `1` — general/runtime error (caught exception, failed operation)
- `2` — usage/argument error (bad flags, missing required argument)

`GetOptions` does **not** exit automatically on parse failure — it only
returns a false value. The script must detect that return value itself and
exit `2`:

```perl
GetOptions(\%opt, 'name=s', 'shout', 'json', 'no_input|no-input', 'help')
    or do { usage(); exit(2); };

if (!defined $opt{name} && !$opt{no_input}) {
    # interactive prompt path, not a usage error
}
elsif (!defined $opt{name}) {
    print STDERR "error: --name is required\n";
    exit(2);
}

my $result = eval { App::MyTool::Core::build_greeting(%opt) };
if (my $err = $@) {
    print STDERR "error: $err";
    exit(1);
}
exit(0);
```

Wrap every call into `Core.pm` in `eval { }` and treat a caught `$@` as exit
`1`; treat every argument-shape problem — whether caught by `GetOptions`
itself or detected afterward (mutually exclusive flags, missing required
value) — as exit `2`.

## 6. `--json` output and `--no-input`

`--json`: use `JSON::PP` (core-shipped since Perl 5.14, no dependency
declaration needed) to encode the same result `Core.pm` returns for the
text-rendering path — never build a separate JSON-shaped return value in
`Core.pm` itself:

```perl
use JSON::PP qw(encode_json);

my $result = App::MyTool::Core::build_greeting(%opt);
if ($opt{json}) {
    print encode_json({ text => $result->{text} }), "\n";
}
else {
    print "$result->{text}\n";
}
```

`--no-input`: guard every interactive `prompt`-style read behind it. If a
required value is missing and `--no-input` is set, exit `2` immediately
instead of blocking on `STDIN` — a bare prompt hangs CI and piped
invocations:

```perl
sub prompt_or_die {
    my ($message, $opt) = @_;
    if ($opt->{no_input}) {
        print STDERR "error: missing required input and --no-input set\n";
        exit(2);
    }
    print "$message: ";
    my $answer = <STDIN>;
    chomp $answer;
    return $answer;
}
```

Never call `<STDIN>` directly from `script/mytool` without first checking
`$opt{no_input}` — every read has to route through a guard like the one
above.

## 7. stdout/stderr discipline

Route results and data — the thing a downstream `|` pipe consumes — to
stdout via `print`. Route diagnostics, progress, and log lines to `STDERR`
via `print STDERR` or `warn`:

```perl
print "$result->{text}\n";                  # stdout: data
print STDERR "fetching...\n";               # stderr: progress/diagnostic
warn "cache miss, refetching\n";             # stderr: diagnostic (warn always goes to STDERR)
```

Never let a diagnostic or progress line land on stdout — it corrupts
`--json` output and breaks piping into `jq`/`grep`/downstream tools. Prefer
`warn` over bare `print STDERR` for anything log-shaped, since `warn`
guarantees `STDERR` even if a caller has fiddled with filehandles.

## 8. Distribution

`minil new App-MyTool` generates the standard `cpanfile`, `minil.toml`
(Minilla's own config, replacing a hand-written `Makefile.PL`/`dist.ini`),
and computes `META.json` at build time — do not hand-write `META.json`.

Declare runtime and test dependencies in `cpanfile`:

```
requires 'perl', '5.014';
on test => sub {
    requires 'Test::More', '0.98';
    requires 'Test::Script';
    requires 'Test::Differences';
    requires 'Capture::Tiny';
};
```

Build and publish:

```bash
minil test              # run t/
minil dist               # build the release tarball
minil release             # upload to PAUSE/CPAN (prompts for PAUSE creds)
# or, from the built tarball directly:
cpan-upload App-MyTool-0.01.tar.gz
```

## 9. Snapshot testing `--help`

`Test::Script::script_runs()` is an assertion function — it checks the
script's exit status itself but does not return captured output, so it
cannot be used to diff stdout against a golden file. Use `Capture::Tiny`
to actually capture stdout/stderr from the subprocess, keep
`script_compiles` for the syntax check, and diff the captured stdout
against a golden file in `t/data/` with `Test::Differences::eq_or_diff()`
so failures show a readable line-level diff instead of a single boolean
mismatch:

```perl
use Test::More;
use Test::Script;
use Test::Differences;
use Capture::Tiny qw(capture);

script_compiles('script/mytool');

my ($stdout, $stderr, $exit) = capture {
    system($^X, 'script/mytool', '--help');
};
is($exit >> 8, 0, 'script/mytool --help exits 0');

my $golden = do {
    local $/;
    open my $fh, '<', 't/data/help.txt' or die $!;
    <$fh>;
};

eq_or_diff($stdout, $golden, 'help text matches golden file');

done_testing;
```

Generate `t/data/help.txt` once by running `script/mytool --help > t/data/help.txt`
and committing it; regenerate it deliberately (same command) whenever the
`usage()` text intentionally changes, never as a side effect of an unrelated
change.

## 10. Minimal worked example

Illustrative only — the actual generation is always freeform from this
reference, never a copy-paste of this exact code:

```perl
# lib/App/MyTool/Core.pm — no CLI imports, REPL-callable
package App::MyTool::Core;
use strict;
use warnings;

sub build_greeting {
    my (%opt) = @_;
    die "name must not be empty\n" unless length $opt{name};
    my $text = "Hello, $opt{name}!";
    $text = uc $text if $opt{shout};
    return { text => $text };
}

1;
```

```perl
#!/usr/bin/env perl
# script/mytool — thin Getopt::Long entry point
use strict;
use warnings;
use Getopt::Long qw(GetOptions);
use JSON::PP qw(encode_json);
use App::MyTool::Core;

my %opt = (shout => 0, json => 0, no_input => 0);
GetOptions(\%opt, 'name=s', 'shout', 'json', 'no_input|no-input', 'help')
    or exit(2);

if ($opt{help}) {
    print "Usage: mytool --name NAME [--shout] [--json] [--no-input]\n";
    exit(0);
}

my $result = eval { App::MyTool::Core::build_greeting(%opt) };
if (my $err = $@) {
    print STDERR "error: $err";
    exit(1);
}

print $opt{json} ? encode_json({ text => $result->{text} }) . "\n"
                  : "$result->{text}\n";
exit(0);
```
