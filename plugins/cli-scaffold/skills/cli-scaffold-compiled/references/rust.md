# Rust CLI generation reference

Pin every Rust CLI generated by this skill to the idioms below. Do not
invent alternative shapes — regenerate the same structure every run.

## 1. Framework

Default to **clap** with derive macros. Never generate a hand-rolled
`std::env::args()` parser or a builder-style (non-derive) clap setup — derive
is the only supported shape.

Install:

```bash
cargo add clap --features derive
```

Minimal invocation shape:

```rust
use clap::Parser;

/// Short one-line description shown in --help
#[derive(Parser)]
#[command(name = "mytool", version, about, long_about = None)]
struct Cli {
    /// Name to greet
    #[arg(value_name = "NAME")]
    name: String,

    /// Emit output in uppercase
    #[arg(long)]
    shout: bool,
}
```

`#[command(...)]` always carries `name`, `version` (pulls from
`Cargo.toml`'s `version` field automatically), and `about` (pulls the crate's
doc comment or `Cargo.toml` `description`). `#[arg(...)]` attributes derive
help text from the field's doc comment — write one `///` line per field, not
a separate `help = "..."` string, so there is exactly one source of truth for
each flag's description.

## 2. Project layout

Generate this exact tree:

```
mytool/
├── Cargo.toml
├── README.md
├── src/
│   ├── lib.rs        # business logic, zero clap imports
│   └── main.rs        # thin binary: Cli struct, parse, dispatch
└── tests/
    └── cli.rs          # assert_cmd + insta integration tests
```

`Cargo.toml` must declare both a library and a binary target explicitly:

```toml
[package]
name = "mytool"
version = "0.1.0"
edition = "2021"

[lib]
name = "mytool"
path = "src/lib.rs"

[[bin]]
name = "mytool"
path = "src/main.rs"

[dependencies]
clap = { version = "4", features = ["derive"] }
```

`lib.rs` rule (non-negotiable): every public function/struct in it must
compile and be unit-testable via `cargo test --lib` with zero `clap` in
scope. If a function needs a `clap`-derived type to make sense, it belongs in
`main.rs`, not `lib.rs`.

`main.rs` rule: only `Cli::parse()`, dispatch into `lib.rs` functions, output
formatting (text vs `--json`), and exit-code mapping. No business rules — those
live in `lib.rs` so they're testable without spawning the compiled binary.

## 3. Help text and completions

clap derives `--help` structure entirely from the `Cli` struct: doc comments
become descriptions, field order becomes argument order. It always produces:

```
Usage: mytool [OPTIONS] <NAME>

Arguments:
  <NAME>  Name to greet

Options:
      --shout    Emit output in uppercase
  -h, --help     Print help
  -V, --version  Print version
```

Subcommand apps (`#[derive(Subcommand)] enum Commands`) add a `Commands:`
section above `Options:`, one line per variant, taken from that variant's doc
comment. Rely on this auto-generated structure — never hand-write a `--help`
string.

Shell completions are not built into clap core; generate them with the
companion `clap_complete` crate via a hidden subcommand, not a build script
(build-script-generated completions can't target the user's actual shell
choice at install time):

```bash
cargo add clap_complete
```

```rust
use clap::CommandFactory;
use clap_complete::{generate, Shell};

#[derive(clap::Subcommand)]
enum Commands {
    /// Generate shell completions (hidden from --help)
    #[command(hide = true)]
    GenerateCompletions { shell: Shell },
}

// in main(), after matching Commands::GenerateCompletions { shell }:
generate(shell, &mut Cli::command(), "mytool", &mut std::io::stdout());
```

Document the end-user invocation for each shell:

```bash
mytool generate-completions bash > /usr/local/etc/bash_completion.d/mytool
mytool generate-completions zsh  > "${fpath[1]}/_mytool"
mytool generate-completions fish > ~/.config/fish/completions/mytool.fish
mytool generate-completions powershell > mytool.ps1
```

## 4. NO_COLOR-aware output

clap's own help/error rendering (via its internal `anstream`/`anstyle`
usage) already detects the `NO_COLOR` environment variable and the
no-color.org convention automatically — no code is required for `--help` and
argument-error output to respect it. If the generated CLI also uses
`owo-colors` or `anstream` directly for its own styled output (progress,
diagnostics), those crates honor `NO_COLOR` the same way by default; do not
add custom detection logic on top.

Verify by running the built binary with and without the variable set and
diffing for ANSI escapes:

```bash
mytool --help | cat -v            # baseline, TTY heuristics may still color it
NO_COLOR=1 mytool --help | cat -v  # must contain no ^[[ escape sequences
```

To force color off unconditionally regardless of environment (e.g. for a
`--no-color` flag distinct from `NO_COLOR`), set it explicitly on the
`Command`:

```rust
use clap::{ColorChoice, CommandFactory};

Cli::command().color(ColorChoice::Never);
```

Only add this when the generated CLI needs its own `--no-color` flag
independent of the environment variable — otherwise let clap's automatic
`NO_COLOR` detection stand alone.

## 5. Exit codes

Frozen cross-paradigm contract — every generated CLI, regardless of
language, must honor this:

- `0` — success
- `1` — general/runtime error (caught error, failed operation)
- `2` — usage/argument error (bad flags, missing required argument)

clap already exits with code `2` automatically on argument-parsing failure
(unknown flag, missing required argument, wrong type) — this matches the
contract natively. Do not add manual remapping for that case; contrast this
with .NET's `System.CommandLine`, whose default parse-failure exit code does
*not* match this contract and must be remapped explicitly — Rust needs no
equivalent shim.

Application code only needs to cover `0` and `1` explicitly, via
`std::process::exit`:

```rust
fn main() {
    let cli = Cli::parse();
    match mytool::run(&cli.name, cli.shout) {
        Ok(output) => println!("{output}"),
        Err(err) => {
            eprintln!("error: {err}");
            std::process::exit(1);
        }
    }
}
```

For a usage error detected inside application logic (valid at parse time,
invalid in combination — e.g. two mutually exclusive flags both set), exit
`2` explicitly rather than `1`, since it is still a usage error clap's parser
could not catch on its own:

```rust
if cli.shout && cli.quiet {
    eprintln!("error: --shout and --quiet are mutually exclusive");
    std::process::exit(2);
}
```

## 6. `--json` output and `--no-input`

`--json` flag: keep exactly one rendering branch difference in `main.rs` —
call the same `lib.rs` function regardless, then switch how the result is
printed. Add `serde` (with the `derive` feature) and `serde_json` as
dependencies:

```bash
cargo add serde --features derive
cargo add serde_json
```

```rust
#[derive(serde::Serialize)]
struct Status {
    state: String,
}

if cli.json {
    println!("{}", serde_json::to_string(&status)?);
} else {
    println!("status: {}", status.state);
}
```

`--no-input` flag: guard every interactive `std::io::stdin()` read behind it
instead of silently defaulting. If a required value is missing and
`--no-input` is set, exit `2` (usage error — the invocation was incomplete
for non-interactive use):

```rust
let name = match cli.name {
    Some(n) => n,
    None if cli.no_input => {
        eprintln!("error: --name is required with --no-input");
        std::process::exit(2);
    }
    None => {
        let mut buf = String::new();
        print!("Project name: ");
        std::io::stdin().read_line(&mut buf)?;
        buf.trim().to_string()
    }
};
```

Never call `std::io::stdin().read_line(...)` without first checking
`no_input` — an unguarded read hangs a non-interactive invocation (CI, piped
stdin) instead of failing fast.

## 7. stdout/stderr discipline

Route results and data — the thing a downstream `|` pipe consumes — to
stdout via `println!`/`print!`. Route diagnostics, progress, and log lines to
stderr via `eprintln!`/`eprint!`:

```rust
println!("{result}");              // stdout: data
eprintln!("fetching...");          // stderr: progress/diagnostic
```

For anything beyond a one-line diagnostic, wire the `tracing` or `log` crate
to a stderr subscriber instead of stacking `eprintln!` calls. Never let a log
call or progress message land on stdout — it corrupts `--json` output and
breaks piping into `jq`/`grep`/downstream tools.

## 8. Distribution

Publish to crates.io via `cargo publish`; the lib+bin split in `Cargo.toml`
(Section 2) is what makes the crate both `cargo add`-able as a library and
`cargo install`-able as a binary from the same package. Document the
end-user install command:

```bash
cargo install mytool
```

Document publishing:

```bash
cargo login                # one-time, stores crates.io API token
cargo publish --dry-run    # verify packaging before the real publish
cargo publish
```

## 9. Snapshot testing `--help`

Use `insta` + `assert_cmd` to pin `--help` output. `assert_cmd` invokes the
already-built binary; `insta` captures and diffs its stdout as a snapshot.

```bash
cargo add --dev assert_cmd insta
```

Generate `tests/cli.rs`:

```rust
use assert_cmd::Command;

#[test]
fn help_snapshot() {
    let output = Command::cargo_bin("mytool")
        .unwrap()
        .arg("--help")
        .output()
        .unwrap();
    let stdout = String::from_utf8(output.stdout).unwrap();
    insta::assert_snapshot!(stdout);
}
```

Review and accept the first-run snapshot with `cargo insta review` (requires
`cargo install cargo-insta` once), then run plain `cargo test` in CI so any
unreviewed `--help` drift fails the build instead of silently passing.

## 10. Minimal worked example

Illustrative only — the actual generation is always freeform from this
reference, never a copy-paste of this exact code:

```rust
// src/lib.rs — no clap imports, unit-testable via `cargo test --lib`
pub fn build_greeting(name: &str, shout: bool) -> Result<String, String> {
    if name.is_empty() {
        return Err("name must not be empty".to_string());
    }
    let text = format!("Hello, {name}!");
    Ok(if shout { text.to_uppercase() } else { text })
}
```

```rust
// src/main.rs — thin clap entry point
use clap::Parser;

#[derive(Parser)]
#[command(name = "mytool", version, about)]
struct Cli {
    /// Name to greet
    name: String,
    /// Emit output in uppercase
    #[arg(long)]
    shout: bool,
}

fn main() {
    let cli = Cli::parse();
    match mytool::build_greeting(&cli.name, cli.shout) {
        Ok(text) => println!("{text}"),
        Err(err) => {
            eprintln!("error: {err}");
            std::process::exit(1);
        }
    }
}
```
