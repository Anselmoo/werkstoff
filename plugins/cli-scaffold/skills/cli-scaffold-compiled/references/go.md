# Go CLI generation reference

Pin every Go CLI generated by this skill to the idioms below. Do not invent
alternative shapes — regenerate the same structure every run.

## 1. Framework

Default to **cobra**, scaffolded via **cobra-cli** — the de facto production
standard (kubectl, gh, hugo all use it). Never hand-roll flag parsing with
bare `flag` or reach for an alternative framework (urfave/cli, kingpin)
unless the user explicitly asks for zero dependencies.

Install the scaffolding tool once, then use it to generate the tree and every
command:

```bash
go install github.com/spf13/cobra-cli@latest
cobra-cli init --pkg-name <module>
cobra-cli add <name>
```

`cobra-cli init` creates `cmd/root.go` and wires `main.go` to call
`cmd.Execute()`. `cobra-cli add <name>` appends `cmd/<name>.go` with a
pre-registered `*cobra.Command`. Rely on this generated shape rather than
writing `*cobra.Command` literals by hand from scratch.

Minimal command shape (what `cobra-cli add` produces, trimmed):

```go
var greetCmd = &cobra.Command{
	Use:   "greet [name]",
	Short: "Print a greeting",
	Args:  cobra.ExactArgs(1),
	RunE: func(cmd *cobra.Command, args []string) error {
		return runGreet(cmd, args[0])
	},
}

func init() {
	rootCmd.AddCommand(greetCmd)
}
```

Always use `RunE` (returns `error`), never `Run` (returns nothing) — `RunE`
is the only shape that lets an error propagate to `main.go` for exit-code
mapping (see §5). `RunE`'s body does argument/flag plumbing only, then
delegates immediately into `internal/<app>/`.

## 2. Project layout

Generate this exact tree:

```
myapp/
├── go.mod
├── go.sum
├── README.md
├── cmd/
│   └── myapp/
│       ├── main.go        # thin entry point: builds root *cobra.Command, os.Exit
│       ├── root.go         # rootCmd (persistent flags: --json, --no-input)
│       └── greet.go        # one file per subcommand, RunE delegates into internal/
├── internal/
│   └── myapp/
│       ├── core.go         # business logic, zero cobra/CLI imports
│       └── core_test.go
└── testdata/
    └── help.golden          # golden file for --help snapshot testing
```

`cmd/<app>/` holds cobra command wiring — either flat as above (what
`cobra-cli init`/`cobra-cli add` generate by default) or, once the command
count grows past roughly half a dozen, moved into a dedicated
`cmd/<app>/commands/` sub-package with `main.go` reduced to importing it and
calling `commands.Execute()`. State this precisely when scaffolding: default
to the flat `cmd/<app>/*.go` layout cobra-cli itself produces; only introduce
`commands/` when the user's command count justifies the extra package
boundary.

`internal/<app>/` rule (non-negotiable): every exported function/type in it
must compile and be unit-testable via `go test ./internal/...` with zero
imports of `github.com/spf13/cobra` or any flag-parsing package. If a
function needs a `*cobra.Command` to make sense, it belongs in `cmd/`, not
`internal/`.

## 3. Help text and completions

Cobra auto-generates `--help` with this fixed structure — never hand-write a
help string:

```
Usage:
  myapp greet [name] [flags]

Flags:
  -h, --help   help for greet

Global Flags:
      --json       output as JSON
      --no-input   disable interactive prompts
```

For the root command with subcommands, cobra additionally renders an
`Available Commands:` section listing each subcommand's `Short` description.
Rely on this generated layout; only customize `Short`/`Long`/`Example` fields
on each `*cobra.Command`, never the surrounding structure.

`cobra-cli init` scaffolds a built-in `completion` subcommand automatically
(bash, zsh, fish, PowerShell). Document the exact install command for the
end user's shell:

```bash
myapp completion zsh > "${fpath[1]}/_myapp"   # zsh
myapp completion bash > /etc/bash_completion.d/myapp   # bash
myapp completion fish > ~/.config/fish/completions/myapp.fish   # fish
```

Do not hand-write a completion script — `completion` ships as part of the
`cobra-cli init` scaffold with zero extra code.

## 4. NO_COLOR-aware output

Check `os.Getenv("NO_COLOR")` explicitly before invoking any coloring
library, even though `fatih/color` (the standard choice here) already
respects `NO_COLOR` automatically as of recent versions by disabling color
output when the variable is set to any non-empty value. State this
precisely rather than assuming manual plumbing is required — but when a
stricter, verifiable guarantee is needed (e.g. the CLI must never emit ANSI
codes when piped, independent of the library's own detection timing), force
it explicitly:

```go
import (
	"os"

	"github.com/fatih/color"
)

func init() {
	if os.Getenv("NO_COLOR") != "" {
		color.NoColor = true
	}
}
```

Set this in `cmd/<app>/main.go` (or `root.go`'s `init()`), before any command
runs — never inside `internal/`, which must not import a coloring library at
all.

## 5. Exit codes

Frozen cross-paradigm contract — every generated CLI, regardless of
language, must honor this:

- `0` — success
- `1` — general/runtime error (caught exception, failed operation)
- `2` — usage/argument error (bad flags, missing required argument)

Call `os.Exit(N)` only from `main.go`, never from inside `internal/<app>/` —
functions there return `error` values (or a typed error implementing an
`ExitCoder`-style interface if the distinction between "runtime error" and
"usage error" must survive the return trip); `main.go` is the single place
that translates an `error` into a process exit code.

Cobra's own default behavior on an argument-parsing failure (unknown flag,
`cobra.ExactArgs` mismatch) returns exit code `1`, not `2` — this diverges
from this plugin's frozen contract and must be remapped explicitly. Contrast
this with Rust/clap, whose default argument-error exit code is already `2`
and needs no remapping. Do the remap in `main.go` by setting
`SilenceUsage`/`SilenceErrors` on the root command and handling the returned
error yourself instead of letting `cmd.Execute()`'s default error path exit
for you:

```go
// cmd/myapp/main.go
func main() {
	rootCmd.SilenceUsage = true
	rootCmd.SilenceErrors = true
	if err := rootCmd.Execute(); err != nil {
		fmt.Fprintln(os.Stderr, "error:", err)
		if isUsageError(err) {
			os.Exit(2)
		}
		os.Exit(1)
	}
}
```

`isUsageError` distinguishes a usage error (bad flag combination, missing
required argument) from a runtime error (e.g. network failure) — either by
sentinel error wrapping (`errors.Is`) or a small custom error type returned
from `RunE`/`internal/`. Do not rely on cobra's own `Args` validators alone
to reach exit code `2`; they land on `1` without this explicit remap.

## 6. `--json` output and `--no-input`

Register `--json` and `--no-input` as **persistent flags** on the root
command so every subcommand inherits them without redeclaring:

```go
// cmd/myapp/root.go
func init() {
	rootCmd.PersistentFlags().Bool("json", false, "output as JSON")
	rootCmd.PersistentFlags().Bool("no-input", false, "disable interactive prompts")
}
```

`--json`: call the same `internal/<app>/` function regardless of the flag,
then branch only on how the result is rendered, using `encoding/json`:

```go
result := core.Greet(name)
if jsonOut, _ := cmd.Flags().GetBool("json"); jsonOut {
	b, err := json.Marshal(result)
	if err != nil {
		return err
	}
	cmd.Println(string(b))
} else {
	cmd.Println(result.Text)
}
```

`--no-input`: guard every interactive `bufio.NewReader(os.Stdin)` read
behind this flag. If a required value is missing and `--no-input` is set,
return a usage error mapped to exit code `2` (see §5) instead of blocking on
stdin:

```go
noInput, _ := cmd.Flags().GetBool("no-input")
if name == "" {
	if noInput {
		return usageErrorf("--name is required with --no-input")
	}
	reader := bufio.NewReader(os.Stdin)
	fmt.Print("Project name: ")
	name, _ = reader.ReadString('\n')
	name = strings.TrimSpace(name)
}
```

Never call `bufio.NewReader(os.Stdin).ReadString(...)` without first checking
`--no-input` — an unguarded read hangs a non-interactive invocation (CI,
piped stdin) instead of failing fast.

## 7. stdout/stderr discipline

Route results and data — the thing a downstream `|` pipe consumes — to
stdout via `cmd.Println`/`fmt.Println`. Route diagnostics, progress, and log
lines to stderr via `cmd.PrintErrln`/`fmt.Fprintln(os.Stderr, ...)`:

```go
cmd.Println(result.Text)                       // stdout: data
fmt.Fprintln(os.Stderr, "fetching...")          // stderr: progress/diagnostic
```

Prefer `cmd.Println`/`cmd.PrintErrln` over bare `fmt.Println` inside
`RunE` — cobra routes them through the command's configured `OutOrStdout()`/
`ErrOrStderr()`, which keeps output redirectable in tests without swapping
global `os.Stdout`. Never let a log call or progress message land on
stdout — it corrupts `--json` output and breaks piping into `jq`/`grep`/
downstream tools.

## 8. Distribution

Document `go install` as the end-user install command, resolving from the
module path and a tagged version:

```bash
go install github.com/<org>/<repo>/cmd/myapp@latest
```

`go install` resolves versions from the module proxy's tag list, so tagging
every release is required, not optional:

```bash
git tag v1.0.0
git push --tags
```

For binaries distributed outside of `go install` — Homebrew formulas,
`.deb`/`.rpm` packages, prebuilt archives attached to a GitHub release — use
GoReleaser to build and publish them from the same git tag; this is the
idiomatic Go counterpart to the shell paradigm's Homebrew/apt distribution
channel, so cross-reference that section rather than duplicating packaging
detail here.

## 9. Snapshot testing `--help`

Use a `testdata/` golden file plus a `-update` flag to pin `--help` output.
Capture the command's output via `cmd.SetOut(&buf)`, invoke `Execute()` with
`["--help"]`, and compare against the stored golden file:

```go
// cmd/myapp/help_test.go
var update = flag.Bool("update", false, "update golden files")

func TestHelp(t *testing.T) {
	var buf bytes.Buffer
	rootCmd.SetOut(&buf)
	rootCmd.SetArgs([]string{"--help"})
	if err := rootCmd.Execute(); err != nil {
		t.Fatal(err)
	}

	golden := "testdata/help.golden"
	if *update {
		os.WriteFile(golden, buf.Bytes(), 0o644)
	}
	want, err := os.ReadFile(golden)
	if err != nil {
		t.Fatal(err)
	}
	if !bytes.Equal(buf.Bytes(), want) {
		t.Errorf("help output mismatch (run with -update to regenerate):\n%s",
			cmp.Diff(string(want), buf.String()))
	}
}
```

Regenerate the golden file after any intentional `--help` text change:

```bash
go test -run TestHelp -update
```

Run plain `go test ./...` (without `-update`) in CI so any unreviewed
`--help` drift fails the build instead of silently passing.

## 10. Minimal worked example

Illustrative only — the actual generation is always freeform from this
reference, never a copy-paste of this exact code:

```go
// internal/myapp/core.go — no cobra/CLI imports, plain-Go testable
package myapp

import "fmt"

type Greeting struct {
	Text string `json:"text"`
}

func Greet(name string, shout bool) (Greeting, error) {
	if name == "" {
		return Greeting{}, fmt.Errorf("name must not be empty")
	}
	text := fmt.Sprintf("Hello, %s!", name)
	if shout {
		text = strings.ToUpper(text)
	}
	return Greeting{Text: text}, nil
}
```

```go
// cmd/myapp/main.go — thin cobra entry point
func main() {
	rootCmd.PersistentFlags().Bool("json", false, "output as JSON")

	var shout bool
	greetCmd := &cobra.Command{
		Use:  "greet [name]",
		Args: cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			g, err := myapp.Greet(args[0], shout)
			if err != nil {
				return err
			}
			if jsonOut, _ := cmd.Flags().GetBool("json"); jsonOut {
				b, _ := json.Marshal(g)
				cmd.Println(string(b))
			} else {
				cmd.Println(g.Text)
			}
			return nil
		},
	}
	greetCmd.Flags().BoolVar(&shout, "shout", false, "uppercase the greeting")
	rootCmd.AddCommand(greetCmd)

	rootCmd.SilenceUsage, rootCmd.SilenceErrors = true, true
	if err := rootCmd.Execute(); err != nil {
		fmt.Fprintln(os.Stderr, "error:", err)
		os.Exit(1)
	}
}
```
