# .NET CLI generation reference

Pin every .NET CLI generated by this skill to the idioms below. Do not
invent alternative shapes — regenerate the same structure every run.

## 1. Framework

Use **System.CommandLine** (Microsoft's official modern CLI parsing
library) — no other parser. Add it to the `<App>.Cli.csproj` host project
only, never to `<App>.Core.csproj`:

```bash
dotnet add <App>.Cli/<App>.Cli.csproj package System.CommandLine
```

System.CommandLine reached a stable 2.0.0 GA release in November 2025 (now
at 2.0.10+) — do not add `--prerelease` or pin a `beta`/`rc` version; that
guidance is stale as of any check after GA. Pin an exact stable version in
the `.csproj` (`Version="2.0.10"` or later, no wildcard) so `dotnet restore`
is reproducible across machines. Verify the current latest stable version
against `https://api.nuget.org/v3-flatcontainer/system.commandline/index.json`
before pinning, since this package moves fast.

Minimal `RootCommand`/`Option<T>` shape — current (2.0 GA) API, built on
`SetAction` and `Parse(...).Invoke(...)`, not the older handler-builder
API from the pre-GA beta series:

```csharp
using System.CommandLine;

var nameOption = new Option<string>("--name", "-n") { Description = "Your name" };

var rootCommand = new RootCommand("Sample command-line app");
rootCommand.Options.Add(nameOption);

rootCommand.SetAction(parseResult =>
{
    string? name = parseResult.GetValue(nameOption);
    Console.Out.WriteLine($"Hello, {name ?? "World"}!");
    return 0;
});

return rootCommand.Parse(args).Invoke();
```

Subcommands are `Command` instances added via `rootCommand.Subcommands.Add(subCommand)`,
each with its own `Options`/`Arguments`/`SetAction`.

## 2. Project layout

Generate this exact tree — a two-project solution plus a mirrored test tree:

```
<App>/
├── <App>.sln
├── <App>.Core/
│   ├── <App>.Core.csproj
│   └── <Domain>.cs            # plain public classes/methods, zero System.CommandLine refs
├── <App>.Cli/
│   ├── <App>.Cli.csproj       # references <App>.Core, packs as a dotnet tool
│   └── Program.cs             # RootCommand/subcommand/option wiring only
└── tests/
    ├── <App>.Core.Tests/
    │   ├── <App>.Core.Tests.csproj
    │   └── <Domain>Tests.cs
    └── <App>.Cli.Tests/
        ├── <App>.Cli.Tests.csproj
        ├── HelpSnapshotTests.cs
        └── HelpSnapshotTests.Help_output_matches_snapshot.verified.txt
```

`<App>.Core.csproj` rule (non-negotiable): no `<PackageReference>` to
`System.CommandLine` and no `using System.CommandLine` anywhere under this
project. Every class/method must be callable and unit-testable from a
plain xUnit test with no CLI parser loaded.

`Program.cs` rule: only `RootCommand`/`Command`/`Option` construction,
calling into `<App>.Core` types, output formatting (text vs `--json`), and
exit-code mapping. No business conditionals — those live in `<App>.Core`.

## 3. Help text and completions

System.CommandLine auto-generates `--help` with this structure — rely on
it, never hand-write a help string:

```
Description:
  <RootCommand description>

Usage:
  <app> [command] [options]

Commands:
  <subcommand>  <subcommand description>

Arguments:
  <arg>  <argument description>

Options:
  -n, --name <name>  Your name
  --version          Show version information
  -?, -h, --help     Show help and usage information
```

Shell completions are built on **dotnet-suggest** and ship free with any
System.CommandLine app once the tool is installed and registered — do not
hand-roll a completion script. Document registration once per shell:

```bash
dotnet tool install -g dotnet-suggest

# bash
dotnet suggest script bash >> ~/.bashrc

# zsh
dotnet-suggest script zsh > ~/.dotnet-suggest-shim.zsh
echo '. ~/.dotnet-suggest-shim.zsh' >> ~/.zshrc

# PowerShell
dotnet suggest script powershell >> $PROFILE
```

`dotnet-suggest` discovers any System.CommandLine-based executable on
`PATH` automatically — no per-app registration code is needed in
`Program.cs`.

## 4. NO_COLOR-aware output

The .NET BCL has no built-in `NO_COLOR` convention — `Console` and
System.CommandLine's own help renderer do not check it automatically.
Check it explicitly, once, near the top of `Program.cs`:

```csharp
bool noColor = !string.IsNullOrEmpty(Environment.GetEnvironmentVariable("NO_COLOR"));
```

If the generated CLI has no extra styling beyond plain `Console` output,
this flag mainly guards manual ANSI escape sequences — strip them when
`noColor` is true. If `Spectre.Console` is already a dependency (only add
it if the user asked for rich output), wire the flag into its color
system explicitly rather than trusting autodetection:

```csharp
AnsiConsole.Profile.Capabilities.ColorSystem = noColor ? ColorSystem.NoColors : AnsiConsole.Profile.Capabilities.ColorSystem;
```

## 5. Exit codes

Frozen cross-paradigm contract — every generated CLI, regardless of
language, must honor this:

- `0` — success
- `1` — general/runtime error (caught exception, failed operation)
- `2` — usage/argument error (bad flags, missing required argument)

Command handlers return `int` from `SetAction(Func<ParseResult, int>)` (or
`Task<int>` for async), which `Parse(...).Invoke(...)` surfaces directly
as the process exit code.

**Deviation to correct for:** unlike Click/Cobra-family frameworks,
System.CommandLine's out-of-the-box behavior does **not** already match
this contract — a parse failure (unknown option, missing required
argument, wrong type) makes `Invoke()` return `1`, not `2`. Always remap
that default explicitly so parse errors report `2`:

```csharp
ParseResult parseResult = rootCommand.Parse(args);
int exitCode = parseResult.Invoke();
if (exitCode == 1 && parseResult.Errors.Count > 0)
{
    exitCode = 2; // System.CommandLine's default (1) corrected to this contract's usage-error code
}
return exitCode;
```

Application code raised from inside `SetAction` still returns `1` for
runtime errors (caught exceptions, failed operations) and `2` only for
usage errors detected inside application logic (e.g. two mutually
exclusive options both set — valid at parse time, invalid together).

## 6. `--json` output and `--no-input`

`--json` option: one rendering branch in `Program.cs`, same `<App>.Core`
call regardless:

```csharp
var jsonOption = new Option<bool>("--json") { Description = "Emit JSON output" };

rootCommand.SetAction(parseResult =>
{
    var result = MyCore.GetStatus();
    if (parseResult.GetValue(jsonOption))
        Console.Out.WriteLine(System.Text.Json.JsonSerializer.Serialize(result));
    else
        Console.Out.WriteLine($"status: {result.State}");
    return 0;
});
```

`--no-input` option: guard every `Console.ReadLine()` call behind it. If a
required value is missing while `--no-input` is set, fail fast at exit
code `2` instead of blocking on stdin:

```csharp
var noInputOption = new Option<bool>("--no-input") { Description = "Disable interactive prompts" };

rootCommand.SetAction(parseResult =>
{
    string? name = parseResult.GetValue(nameOption);
    bool noInput = parseResult.GetValue(noInputOption);
    if (string.IsNullOrEmpty(name))
    {
        if (noInput)
        {
            Console.Error.WriteLine("error: --name is required with --no-input");
            return 2;
        }
        Console.Out.Write("Project name: ");
        name = Console.ReadLine();
    }
    // ...
    return 0;
});
```

## 7. stdout/stderr discipline

`Console.WriteLine` and `Console.Out.WriteLine` are the same stream by
default, but only `Console.Out` makes the target explicit and stays
correct if a caller redirects `Console.Out`/`Console.Error` independently
(as `InvocationConfiguration.Output`/`.Error` do in tests). Always write
data through `Console.Out`, diagnostics through `Console.Error`:

```csharp
Console.Out.WriteLine(result);                 // stdout: data a `|` pipe consumes
Console.Error.WriteLine("fetching...");         // stderr: progress/diagnostic/log line
```

Never let a log or progress line land on stdout — it corrupts `--json`
output and breaks piping into `jq`/`grep`/downstream tools.

## 8. Distribution

Pack `<App>.Cli.csproj` as a .NET tool:

```xml
<PropertyGroup>
  <PackAsTool>true</PackAsTool>
  <ToolCommandName><App></ToolCommandName>
  <PackageId><App></PackageId>
  <Version>0.1.0</Version>
</PropertyGroup>
```

Document publishing to NuGet:

```bash
dotnet pack <App>.Cli/<App>.Cli.csproj -c Release
dotnet nuget push <App>.Cli/bin/Release/<App>.*.nupkg \
  --api-key <NUGET_API_KEY> --source https://api.nuget.org/v3/index.json
```

Document the end-user install command:

```bash
dotnet tool install -g <App>
```

## 9. Snapshot testing `--help`

Use **Verify.Xunit** to pin `--help` output. Add it to the CLI test
project:

```bash
dotnet add tests/<App>.Cli.Tests/<App>.Cli.Tests.csproj package Verify.Xunit
```

Capture `--help` by invoking the same `RootCommand` in-process with a
redirected `InvocationConfiguration.Output`, then assert against a Verify
snapshot with the exact `await Verify(output)` idiom:

```csharp
using System.CommandLine;
using VerifyXunit;
using Xunit;

[UsesVerify]
public class HelpSnapshotTests
{
    [Fact]
    public async Task Help_output_matches_snapshot()
    {
        var output = new StringWriter();
        var config = new InvocationConfiguration { Output = output };

        RootCommand rootCommand = Program.BuildRootCommand();
        rootCommand.Parse(["--help"]).Invoke(config);

        await Verify(output.ToString());
    }
}
```

On first run this writes `HelpSnapshotTests.Help_output_matches_snapshot.received.txt`;
rename/approve it to `HelpSnapshotTests.Help_output_matches_snapshot.verified.txt`
next to the test file and commit the `.verified.txt` — never the
`.received.txt` — to source control. Subsequent runs diff the live
`--help` output against the committed `.verified.txt` and fail the test
on any unreviewed drift.

## 10. Minimal worked example

Illustrative only — the actual generation is always freeform from this
reference, never a copy-paste of this exact code.

```csharp
// <App>.Core/Greeter.cs — no System.CommandLine imports, plain-testable
namespace App.Core;

public record Greeting(string Text);

public static class Greeter
{
    public static Greeting Build(string name, bool shout = false)
    {
        if (string.IsNullOrWhiteSpace(name))
            throw new ArgumentException("name must not be empty");
        string text = $"Hello, {name}!";
        return new Greeting(shout ? text.ToUpperInvariant() : text);
    }
}
```

```csharp
// <App>.Cli/Program.cs — thin System.CommandLine entry point
using System.CommandLine;
using System.Text.Json;
using App.Core;

var nameArgument = new Argument<string>("name");
var shoutOption = new Option<bool>("--shout");
var jsonOption = new Option<bool>("--json");

var rootCommand = new RootCommand("Greet someone");
rootCommand.Arguments.Add(nameArgument);
rootCommand.Options.Add(shoutOption);
rootCommand.Options.Add(jsonOption);

rootCommand.SetAction(parseResult =>
{
    try
    {
        var greeting = Greeter.Build(
            parseResult.GetValue(nameArgument)!,
            parseResult.GetValue(shoutOption));

        if (parseResult.GetValue(jsonOption))
            Console.Out.WriteLine(JsonSerializer.Serialize(greeting));
        else
            Console.Out.WriteLine(greeting.Text);
        return 0;
    }
    catch (ArgumentException ex)
    {
        Console.Error.WriteLine($"error: {ex.Message}");
        return 1;
    }
});

ParseResult parseResult = rootCommand.Parse(args);
int exitCode = parseResult.Invoke();
return exitCode == 1 && parseResult.Errors.Count > 0 ? 2 : exitCode;
```
